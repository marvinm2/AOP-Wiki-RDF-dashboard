**1. plots/trends_plots.py — Add return type annotation (line 528)**

This ensures safe_plot_execution detects the expected 3-tuple return type on failure and generates `(fallback, fallback, fallback)` instead of a single string. The fallback tuple must match the special-case handling in safe_plot_execution — but note: safe_plot_execution only special-cases `plot_main_graph` for 3-tuples. Add a branch in safe_plot_execution (plots/shared.py) for 3-element string tuples:

In plots/shared.py, inside safe_plot_execution, in the `elif 'tuple' in str(return_type):` branch (around line 864-870), change the logic from:
```python
if 'plot_main_graph' in plot_func.__name__:
    return fallback, fallback, pd.DataFrame()
else:
    return fallback, fallback
```
To:
```python
# Count expected tuple elements from annotation
type_str = str(return_type)
str_count = type_str.count('str')
if str_count >= 3:
    return tuple([fallback] * str_count)
elif 'DataFrame' in type_str:
    return fallback, fallback, pd.DataFrame()
else:
    return fallback, fallback
```

This is more general: it counts how many `str` elements the tuple annotation declares and returns that many fallbacks. For plot_aop_lifetime (tuple[str, str, str]) it returns 3 fallbacks. For plot_main_graph (tuple[str, str, pd.DataFrame]) it hits the DataFrame branch. For standard 2-tuples it returns 2 fallbacks.

**2. app.py — Add content validation before returning success:true (lines 1491-1496)**

Replace the plot_map branch:
```python
elif plot_name in plot_map:
    plot_html = plot_map[plot_name]
    # If it's a callable (lambda function), execute it to generate the plot on-demand
    if callable(plot_html):
        plot_html = plot_html()
    return jsonify({'html': plot_html, 'success': True})
```

With:
```python
elif plot_name in plot_map:
    plot_html = plot_map[plot_name]
    # If it's a callable (lambda function), execute it to generate the plot on-demand
    if callable(plot_html):
        try:
            plot_html = plot_html()
        except Exception as e:
            logger.error(f"Error generating on-demand plot {plot_name}: {e}")
            return jsonify({'error': str(e), 'success': False}), 500

    # Validate content: empty or fallback HTML should not count as success
    if not plot_html or (isinstance(plot_html, str) and len(plot_html.strip()) == 0):
        return jsonify({'error': f'Plot {plot_name} has no data available', 'success': False}), 500
    if isinstance(plot_html, str) and 'Data Unavailable' in plot_html:
        return jsonify({'error': f'Plot {plot_name} failed to load - data unavailable', 'success': False}), 500

    return jsonify({'html': plot_html, 'success': True})
```

This catches three failure modes:
- Exception during callable execution (on-demand plots like aop_completeness_boxplot)
- Empty string from failed unpacking (e.g., plot_aop_lifetime when SPARQL fails)
- Fallback "Data Unavailable" scatter plot from create_fallback_plot (any annotated plot that threw an exception)

The client-side showErrorState() in lazy-loading.js already handles success:false responses with differentiated error cards — no JS changes needed.
